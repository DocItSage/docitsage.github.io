    Спецификация требований к микросервису сбора и анализа вакансий

    Глоссарий

    Термин Определение
    Vacancy Collector Микросервис для автоматического сбора вакансий с API hh.ru.
    Vacancy Analyzer Микросервис для анализа и предоставления данных о вакансиях.
    Celery Распределённая очередь задач для выполнения фоновых операций.
    Redis Брокер сообщений для Celery, хранит задачи в очереди.
    FastAPI Фреймворк для создания API анализа вакансий.
    PostgreSQL Реляционная база данных для хранения вакансий.

    Пояснения к спецификации

    1. Введение описывает какую проблему решает микросервис.
    2. Функциональные требования описывают, что должен делать микросервис (например, сбор вакансий, фильтрация дубликатов).
    3. Нефункциональные требования задают ограничения на производительность, надёжность и безопасность.
    4. Требования к интеграции определяют, как микросервис взаимодействует с внешними и внутренними системами.
    5. Требования к развертыванию описывают инфраструктуру и мониторинг.
    6. Примеры использования показывают, как запускать и использовать микросервис.
    7. Ограничения и риски перечисляют возможные проблемы и их последствия.

    1. Введение

    1.1. Назначение

    Микросервис предназначен для автоматического сбора вакансий с api.hh.ru, их хранения в базе данных и предоставления аналитики пользователям (например, HR-специалистам).

    1.2. Область применения

    • Автоматический сбор вакансий по расписанию.
    • Хранение уникальных вакансий в PostgreSQL.
    • Предоставление API для анализа данных (фильтрация, статистика).

    2. Функциональные требования

    2.1. Сбор вакансий

    ID Требование Детали
    FR-001 Автоматический сбор вакансий Микросервис должен запрашивать
    вакансии с API hh.ru по расписанию (например, каждый час).
    FR-002 Фильтрация дубликатов Перед сохранением вакансии необходимо проверять её уникальность по hh_id.
    FR-003 Хранение "сырых" данных В базе данных должны сохраняться "сырые" данные вакансий в формате JSON.
    FR-004 Поддержка расписания Использовать Celery для запуска задач по расписанию.

    2.2. Хранение данных

    ID Требование Детали
    FR-005 Хранение вакансий Вакансии должны сохраняться в PostgreSQL с полями: hh_id, name, url, salary_from, salary_to, currency, experience, employment, schedule, description, published_at, raw_data.
    FR-006 Индексы для быстрого поиска Добавить индексы для полей hh_id, name, и published_at.

    2.3. Анализ данных

    ID Требование Детали
    FR-007 API для анализа Предоставить FastAPI-эндпоинты для фильтрации вакансий по навыкам, зарплате, региону.
    FR-008 Статистика Реализовать эндпоинты для статистики (например, количество вакансий по регионам, средняя зарплата).
    FR-009 Агрегация данных Поддерживать агрегацию данных по специализациям и регионам.

    3. Нефункциональные требования

    3.1. Производительность

    ID Требование Детали
    NFR-001 Время отклика API Время отклика эндпоинтов FastAPI не должно превышать 500 мс.
    NFR-002 Масштабируемость Микросервис должен поддерживать горизонтальное масштабирование (например, несколько воркеров Celery).

    3.2. Надёжность

    ID Требование Детали
    NFR-003 Обработка ошибок Микросервис должен корректно обрабатывать ошибки API hh.ru (например, повторные запросы при ошибке 429).
    NFR-004 Логирование Все операции должны логироваться (успешные и ошибочные).

    3.3. Безопасность

    ID Требование Детали
    NFR-005 Защита данных Доступ к базе данных должен быть ограничен (например, через .env).

    4. Требования к интеграции

    4.1. Внешние системы

    ID Требование Детали
    INT-001 Интеграция с api.hh.ru Микросервис должен отправлять GET-запросы к https://api.hh.ru/vacancies.
    INT-002 Формат данных Ожидаемый формат ответа от api.hh.ru — JSON.

    4.2. Внутренние зависимости

    ID Требование Детали
    INT-003 Взаимодействие с PostgreSQL Микросервис должен сохранять и извлекать данные из PostgreSQL.
    INT-004 Взаимодействие с Redis Celery должен использовать Redis как брокер сообщений.

    5. Требования к развертыванию

    5.1. Инфраструктура

    ID Требование Детали
    DEP-001 Docker-контейнеризация Микросервис должен разворачиваться через Docker Compose.
    DEP-002 Переменные окружения Конфигурация должна задаваться через .env.

    5.2. Мониторинг

    ID Требование Детали
    DEP-003 Логирование Логи должны сохраняться в файлы и выводиться в stdout.
    DEP-004 Мониторинг задач Celery должен предоставлять информацию о статусе задач.

    7. Ограничения

    Ограничение Детали
    Rate limits API hh.ru api.hh.ru ограничивает количество запросов (например, 100 запросов в минуту). Микросервис должен обрабатывать ошибку 429 Too Many Requests.
    Архивные вакансии Вакансии с archived: true считаются неактуальными и не сохраняются в базе данных.
    Уникальность вакансий Вакансии с одинаковым hh_id не дублируются в базе данных.

    8. Риски

    Риск Последствия Меры по снижению риска
    Изменения в API hh.ru Микросервис может перестать корректно работать, если API hh.ru изменит формат ответа. Регулярно проверять актуальность API и обновлять код при необходимости.
    Блокировка IP-адреса Превышение лимитов запросов к API hh.ru может привести к временной блокировке IP. Реализовать задержки между запросами и обработку ошибок 429.
    Потеря данных Ошибки в базе данных или при сохранении могут привести к потере вакансий. Регулярно создавать резервные копии базы данных и логировать все операции.
